How PagePay ingests OpenStax textbooks alongside Gutendex, wires likes & comments onto books (not slices), and gates education content behind premium — without breaking the existing Read & Earn loop. Grounded in your current client/ + backend/ code.

Reads at the WORK level (not slice) One comment thread per book Free first slice · Premium unlocks the rest Light + Dark mode
Decision 1

Social scope = Work

Likes and comments attach to the book (the parent work in content_catalog), not to individual slices. Matches your existing community pattern. Reuses CommunityLike toggle logic.

Decision 2

Comments per work

One thread per book. The thread is accessible from the book detail screen AND the reader. Slices are read-only contexts — the conversation belongs to the work as a whole.

Decision 3

Education = Premium

First slice of any chapter is always free. Slice 2+ requires premium tier. Casual readers using Read & Earn keep the existing flow unchanged. No regression risk.

Step 1 — What Stays

Home is unchanged for casual readers

The current Home screen already does the job for the Read & Earn audience — greeting, balance chip, streak, resume carousel, category chips, trending feed with native ads every 4th card. Education is added to the catalog and study tabs, not bolted onto Home. Zero regression to the existing flow.

Why we don't put the level grid on Home
  • Your current Home is optimized for the casual reader: greeting + balance + resume + trending. Adding a 6-button level grid would push the resume carousel off-screen on small devices.
  • Education users self-select — they go to Catalog or Study. The Home feed is for "what's new and earnable today."
  • The Catalog tab already has filter infrastructure (category chips). Adding level filters there is a smaller change with higher payoff.

4

Reader — no reward gate, just reading

Education content doesn't earn points

Education content has no pre-read or post-read rewarded ad. There's no 1-minute timer, no scroll-event anti-cheat, no pending_points. The reader becomes a pure content surface. The user reads for understanding, not for points. The bottom of slice #1 nudges them toward premium to continue.

How the reader changes for education content
  • No pre-read ad modal — the existing RewardedAd with visible={preReadOpen} simply never opens for source='openstax' rows.
  • No timer or session — no sessionId, no /api/v1/session/start, no heartbeatRef. Reading is unbounded; the user can take 5 minutes or 50.
  • No pending_points — reading_sessions.pending_points stays NULL. The wallet is never touched for educational reads.
  • Inline ads stay — every 400 chars still gets a native ad (revenue), but ad-watch never credits the user. This is fine: it's an exposure-driven revenue model, not a watch-and-earn model.
  • Slice 1 = end of free content — the bottom of slice 1 shows the premium upsell. Slices 2+ are gated at the book detail level, not in the reader.

Attribution — what it does and why it matters

Legal compliance for OpenStax and beyond
What attribution is

Attribution is the legal requirement of open licenses like Creative Commons (CC BY 4.0). When you use someone else's openly-licensed content commercially, you must credit the original creator visibly. It's the same idea as citing a source in academic work — except it's a legal requirement, not a courtesy.

What the Attribution card does on every OpenStax book:

  • Displays "Source: OpenStax, Rice University" with a clickable link (CC BY §3(a) requires "identification of the creator")
  • Shows the license badge "CC BY 4.0" so users know the book is free to share (CC BY §3(b) requires license indication)
  • Indicates if the work has been modified (CC BY §2(a)(5)(B)) — initially "no modifications" since you slice rather than rewrite
  • Reuses the attribution_text field on content_catalog so the display string is built once at ingest time, not formatted on every reader render
License Commercial Use? Attribution Required? Verdict for PagePay
CC BY 4.0 (OpenStax) ✓ Yes ✓ Yes ✓ YES — with attribution card
Public Domain (Gutenberg) ✓ Yes Optional ✓ YES — no restrictions
CC BY-NC ✗ No ✓ Yes ⚠ REJECTED — ad revenue = commercial
CC BY-SA ✓ Yes ✓ Yes ✓ YES — must share-alike derivatives
Ingestion-time safety net

Every new openstax.py ingest job stamps license_type = 'CC BY 4.0' on every row. A future core.py or ck12.py ingest job must stamp the actual license per paper, and the pipeline must reject any row whose license is not in the allowlist (CC BY, CC BY-SA, Public Domain) before it touches content_catalog. This isn't a UI thing — it's a hard gate at the ingestion boundary so a config error can't ship NC-licensed content to your users.

New tables for social features

Table Fields Purpose
work_likes id, user_id (FK), work_id (FK), created_at One row per like on a work. UNIQUE(user_id, work_id) — same pattern as community_likes. work_id = parent work's id (not slice id).
work_comments id, user_id (FK), work_id (FK), body (Text), parent_comment_id (FK nullable), status (pending/approved/rejected), created_at, updated_at Comments on works. parent_comment_id enables 1-level reply threading. status mirrors community_notes.status for moderation.
work_comment_likes id, user_id (FK), comment_id (FK), created_at Likes on individual comments. UNIQUE(user_id, comment_id).
work_shares id, user_id (FK), work_id (FK), platform (String), created_at Logs each share event for analytics. Platform is "whatsapp" | "instagram" | "twitter" | "copy" | "other".
New API endpoints
  • POST /works/{id}/like — toggle like, returns {liked, count}
  • GET /works/{id}/comments?limit=20&offset=0 — paginated thread, sorted by created_at DESC
  • POST /works/{id}/comments — body: {body: str, parent_comment_id?: int}
  • POST /works/comments/{id}/like — toggle comment like
  • POST /works/{id}/share — body: {platform: str}, logs analytics only
  • GET /works/{id}/social — aggregated {likes, comments, shares, liked_by_me} for cards

Accessibility, edge cases & anti-abuse

What the design plan doesn't usually cover but has to
A11y
Touch targets ≥ 44pt
Like/comment/share buttons all 44pt tall minimum. VoiceOver labels: "Like Calculus Volume 1, 89 likes, double-tap to like."
A11y
Color contrast
All badges (Premium, Source, Level) meet WCAG AA contrast against card background in both schemes. Liked heart (signal orange) passes 4.5:1.
State
Empty states
Book with 0 comments shows "Be the first to discuss this book." Book with 0 likes shows the heart icon outlined, not hidden. Slice 0 of 0 is impossible by data invariant.
State
Loading & error
Social bar shows the existing SkeletonPage pattern while loading. Like failure rolls back optimistic update + shows toast. Comment failure keeps the typed text in the input + retry button.
Abuse
Comment rate limit
3 comments/min, 30/day per user. Reuses your existing slowapi limiter. Prevents spam without breaking real discussion.
Abuse
Cheating-help filter
Comments containing exam question text (matched against StudyMaterial.raw_input) auto-route to status='pending' for moderator review. Critical for exam-prep audience.
i18n
Translation keys
All new copy goes through i18next: social.like, social.comment, social.share, social.comments_count, book.free_preview, book.premium_only, study.continue_chapter.
Network
Offline
Social bar reads from cache. Like/comment actions queue and replay on reconnect (TanStack Query retries). The book list still loads from disk cache via your offline pattern.

Implementation order — what unblocks what
  • Phase A (2 days): 5 new fields on content_catalog + Alembic migration. No new endpoints yet. Lets you write the OpenStax ingest against a stable schema.
  • Phase B (1 week): openstax.py + 5 sample books ingested. Hand-review 20 slices each. Verify attribution strings render correctly.
  • Phase C (3 days): SocialBar + AttributionCard + CommentThread components. work_likes + work_comments + endpoints. useToggleWorkLike hook parallel to useToggleLike.
  • Phase D (1 week): Wire SocialBar into ContentCard + book detail + reader. Wire premium upsell into reader end-of-slice-1. Wire level grid into Catalog.
  • Phase E (optional, +1 week): Study tab bridge — "Turn this chapter into study material" cards. Connects to your existing StudyAsset work.
Risks I'd watch in the first 30 days
  • OpenStax HTML parsing breaks when they update their page structure. Wrap the parser in a defensive test fixture so a single reformat doesn't silently ship empty slices.
  • Comment threading can spiral — start with no replies, add parent_comment_id in Phase 5.2 if demand is real.
  • Premium upsell on every slice 1 end can feel pushy to free users. A/B test against a single mid-chapter upsell or a non-blocking banner.
  • The "free first slice" mechanic only works if you actually lock slice 2+ on the server. Don't trust the client's locked-state to enforce premium — verify the user.tier on the /api/v1/content/{id} response.
