but how does ssv webhook works?

AdMob Server-Side Verification (SSV) webhooks prevent users from hacking your app's code to falsely claim earnings. [1, 2] 
Instead of letting your Android app decide when a user gets paid, Google's servers talk directly to your backend server to confirm the ad was legitimately watched to completion. [1] 
------------------------------
## The 5-Step SSV Communication Flow

[ User Finishes Rewarded Ad ] 
            │
            ▼
   1. Google Servers ──(Sends Webhook + Secret Signature)──► Your Backend Server
                                                                   │
                                                                   ▼
   3. Your Server ◄──(Fetches Public Verification Keys)─── Google Key Server
            │
            ▼
   4. [ Signature Valid? ] ──► YES ──► Credit Wallet 15% (You) / 85% (User)
            │
            ▼
   5. Your Server ──(Returns HTTP 200 OK)──► Google (Stops Retry Loop)


   1. The Trigger: The user finishes watching a 1-minute pre-read or post-read rewarded ad. [1] 
   2. The Ping: Google sends an HTTP GET request to your backend URL. This URL contains vital query string data like the user_id, reward_amount, and a cryptographic signature. [2, 3, 4, 5, 6] 
   3. The Check: Your backend uses Google's public cryptographic keys to verify that the webhook actually came from Google and was not faked by a hacker. [2, 7] 
   4. The Payout: Once the signature is verified, your backend recalculates the 85/15 split and updates the databases.
   5. The Close: Your backend sends back an HTTP 200 OK status to let Google know the message arrived safely. [8, 9] 

------------------------------
## How to Map SSV Data to Your 85/15 Model
When setting up your rewarded ad units in the AdMob dashboard, you must append custom user details to the ad request code inside your app:

// Example Client-Side Code Configurationlet serverSideVerificationOptions = new ServerSideVerificationOptions();
serverSideVerificationOptions.customData = "user_id_99482"; // Tie the ad to a specific user account

When the ad finishes, Google sends the following structured query parameters to your server: [2, 4] 

* custom_data: The unique user ID you sent (user_id_99482).
* reward_amount: The baseline currency token value defined in your AdMob dashboard.
* signature & key_id: Cryptographic tokens used to verify the transaction authenticity. [2, 3, 10] 

------------------------------
## Your Backend Processing Code Logic
Once your server receives and verifies the data payload, your database execution query looks like this:

# Conceptual backend math execution upon valid webhook arrivaldef process_admob_ssv(custom_data, ad_unit_id):
    user_id = custom_data
    
    # 1. Look up live Naira eCPM rates or set token values
    ad_value_naira = 4.20 
    
    # 2. Split according to your 85/15 model
    user_payout = ad_value_naira * 0.85     # ₦3.57 credited to user balance
    platform_cut = ad_value_naira * 0.15    # ₦0.63 kept for business
    
    # 3. Update balances in your secure database
    db.execute("UPDATE users SET wallet = wallet + ? WHERE id = ?", user_payout, user_id)
    db.execute("UPDATE platform_ledger SET balance = balance + ?", platform_cut)

* Implement Idempotency (Replay Protection): Google might accidentally send the exact same webhook twice if a network lag occurs. Your database must log Google's unique transaction_id. If an incoming transaction_id already exists in your system log, discard it immediately so you do not accidentally credit a user twice for one ad watch. [9, 11, 12, 13, 14] 
* Do Not Handle Payouts on the App Interface: Never let the Android or iOS front-end write to your wallet database. Client-side callbacks can be bypassed with runtime injection tools like Frida or Cheat Engine. Rely entirely on the SSV backend webhook to grant earnings. [2, 3]
